# Лабораторна 4: Порівняльний аналіз синхронних та асинхронних side effects

## 1) Що змінилося порівняно з ЛР3

У ЛР3 основний акцент був на CQS/CQRS-структурі: розділення `commands` і `queries`, явні межі читання/запису, краща керованість сервісного шару.

У ЛР4 до цієї структури додано керовані побічні операції (side effects) і два різні способи комунікації з ними:

- **синхронний post-hook** після основної бізнес-операції;
- **асинхронна публікація доменної події** з незалежною обробкою підписниками.

Також введено системний рівень транзакцій:

- `withTransaction(...)` у DB-шарі;
- `AsyncLocalStorage`-контекст для повторного використання того самого DB-клієнта всередині вкладених викликів;
- `getDb()` у DAL, щоб запити автоматично працювали в межах активної транзакції.

Практично це означає, що write-flow тепер контролює не лише зміну даних, а й політику виконання побічних ефектів: які з них блокують успіх операції, а які запускаються "після факту" і не впливають на latency основної відповіді.

## 2) Які побічні операції виділено і чому

### 2.1. Синхронна побічна операція: Anti-spam / Rate Limiter

Виділено окремий компонент `IMessageRateLimiter` (реалізація `DbMessageRateLimiter`), який викликається після створення повідомлення в межах тієї самої транзакції.

**Чому саме синхронно:**

- антиспам-політика є частиною бізнес-інваріанта "користувач не повинен перевищувати ліміт";
- якщо ліміт порушено, система повертає `429` і відкатує транзакцію;
- таким чином не виникає ситуації "повідомлення вже збережено, але ліміт не оновився".

### 2.2. Асинхронна побічна операція: Notification / Realtime Broadcaster

Виділено окремий event-driven компонент:

- подія `MessageWasCreated` публікується після успішного завершення основної операції;
- підписник `message-notification-broadcaster` окремо виконує:
  - realtime broadcast (WebSocket),
  - email-нотифікації для згаданих `@tag`.

**Чому асинхронно:**

- нотифікації не є критичною умовою успіху `POST /messages`;
- вони можуть бути повільними (мережа, SMTP), тому не повинні блокувати API-відповідь;
- тимчасовий збій нотифікацій не повинен ламати основну бізнес-операцію створення повідомлення.

## 3) Порівняння синхронного та асинхронного підходів

### 3.1. Час відповіді API

**Синхронний підхід** збільшує час відповіді на тривалість side-effect операції.  
У нашому випадку write-flow повідомлення включає і DB-оновлення rate-лічильника, тому latency endpoint-а вищий, але передбачуваний.

**Асинхронний підхід** мінімізує latency основної операції.  
API завершується після коміту основної дії; обробка подій (realtime/email) виконується незалежно.

Висновок: для "обов'язкових" інваріантів прийнятний синхронний latency-overhead; для сервісних інтеграцій і повідомлень краще асинхронний режим.

### 3.2. Поведінка при збоях

**Синхронний side effect (rate limit):**

- збій або перевищення ліміту призводить до помилки запиту (`429`);
- транзакція відкочується;
- система залишається строго консистентною щодо інваріанта.

**Асинхронний side effect (notifications):**

- збій підписника не відкочує вже зафіксовану бізнес-операцію;
- подія може бути частково оброблена (наприклад, realtime пройшов, email ні);
- потрібні політики повторів/моніторингу, якщо важлива гарантована доставка.

Висновок: синхронний підхід дає сильні гарантії консистентності; асинхронний підвищує доступність і стійкість API, але переносить складність у шар доставки/обробки подій.

### 3.3. Зв'язаність між компонентами

**Синхронний варіант:**

- handler знає лише інтерфейс `IMessageRateLimiter`, а не конкретний клас;
- coupling помірний: є пряма залежність на контракт, але DIP збережено.

**Асинхронний варіант:**

- handler знає тільки event bus і тип події;
- підписники відокремлені від command-flow, їх можна додавати/міняти без змін основної операції;
- coupling нижчий на рівні бізнес-оркестрації.

Висновок: асинхронна модель краще масштабується за кількістю інтеграцій і side effects.

### 3.4. Складність реалізації та тестування

**Синхронний підхід простіший:**

- одна точка виконання;
- детермінована помилка в тому ж запиті;
- прямі unit/integration тести (вхід -> результат).

**Асинхронний підхід складніший:**

- потрібні події, підписки, bootstrap обробників;
- важче тестувати причинно-часові зв'язки;
- необхідно враховувати eventual consistency, retries, порядок/дублювання подій.

Висновок: синхронний легший для старту; асинхронний складніший в реалізації й операційній підтримці, але дає кращу еволюційність системи.

## 4) Який підхід обрати для production і чому

Для production доцільний **гібридний підхід**:

1. **Синхронно** виконувати side effects, що є частиною критичного бізнес-інваріанта:
   - rate limiting,
   - валідації консистентності,
   - обмеження тарифів/квот.

2. **Асинхронно** виконувати не-критичні та інтеграційні побічні операції:
   - нотифікації,
   - realtime fanout,
   - аналітика,
   - зовнішні інтеграції.

Такий баланс дає:

- контрольовану консистентність там, де це критично;
- швидкий API response time для основних use-case;
- менший coupling і кращу розширюваність для інтеграцій.

## 5) Що буде при подвійній доставці тієї самої події? Чи ідемпотентні обробники

Подвійна доставка подій у real-world системах є нормальним сценарієм (at-least-once semantics), тому обробники мають бути ідемпотентними або "безпечними до повтору".

### Поточний стан у цій реалізації

- **Rate limiter**: побудований як синхронний DB-hook, не як event handler; його логіка атомарна в межах транзакції, але не стосується retry подій.
- **Notification/Realtimе handler**: у поточному вигляді може надіслати дубль realtime/email при повторній доставці події.

Отже, **поточний notification handler не є строго ідемпотентним**.

### Що потрібно для production-ідемпотентності

Рекомендовані кроки:

1. Додати `eventId` у payload/metadata події.
2. Завести таблицю `processed_events(subscriber, event_id, processed_at)` з унікальним ключем.
3. На старті обробки робити "insert-if-not-exists"; якщо запис уже є — пропускати виконання.
4. Для зовнішніх каналів (email/push) використовувати idempotency key, якщо провайдер підтримує.

Після цього повторна доставка не призведе до дубльованих побічних дій.

## Висновок

ЛР4 розширила CQS-архітектуру ЛР3 до практичного керування side effects. Основна цінність змін у тому, що система отримала явну політику: які побічні дії є обов'язковими (синхронний, транзакційний контур), а які мають виконуватись незалежно (асинхронний event-driven контур). Для production найкраще працює поєднання обох підходів із додатковими гарантіями ідемпотентності в подієвих обробниках.
