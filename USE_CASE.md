# USE CASE: Chat API

Документ описує типові сценарії роботи користувачів із системою чату.

## Ролі

- **Гість** — неавторизований користувач.
- **Користувач** — авторизований користувач із власним акаунтом.
- **Власник чату** — користувач, який створив чат.
- **Учасник чату** — користувач, доданий у чат.

## Сценарій 1: Реєстрація та вхід

### Мета

Користувач створює акаунт і отримує токен доступу.

### Передумови

- Користувач не має акаунта з таким самим email або tag.

### Кроки

1. Користувач викликає `POST /auth/register`.
2. Система створює користувача, генерує OTP і токен.
3. Користувач викликає `POST /auth/login`.
4. Система перевіряє дані та повертає токен.

### Результат

- Користувач авторизований і може працювати з чатами.

## Сценарій 2: Створення чату

### Мета

Користувач створює новий чат.

### Передумови

- Користувач має валідний Bearer-токен.

### Кроки

1. Користувач викликає `POST /chats`.
2. Система створює чат.
3. Система автоматично додає автора як учасника.
4. Автор отримує повні права керування чатом.

### Результат

- Новий чат доступний у списку `GET /chats`.

## Сценарій 3: Додавання учасника

### Мета

Власник або адміністратор додає нового учасника до чату.

### Передумови

- Ініціатор має право `memberAdd`.
- Користувач, якого додають, існує.

### Кроки

1. Ініціатор викликає `POST /chats/:chatId/members`.
2. Система перевіряє права ініціатора.
3. Система додає/активує учасника в чаті.
4. Система зберігає передані права учасника.

### Результат

- Новий учасник може читати/писати повідомлення в чаті.

## Сценарій 4: Надсилання та редагування повідомлення

### Мета

Учасник чату публікує повідомлення та, за потреби, редагує його.

### Передумови

- Користувач є активним (не заблокованим) учасником чату.

### Кроки

1. Учасник викликає `POST /chats/:chatId/messages`.
2. Система перевіряє членство.
3. Система створює повідомлення.
4. Автор повідомлення викликає `PATCH /chats/:chatId/messages/:messageId`.
5. Система дозволяє редагування лише автору повідомлення.

### Результат

- Повідомлення збережено/оновлено та доступне через `GET /chats/:chatId/messages`.

## Сценарій 5: Видалення повідомлення

### Мета

Видалити повідомлення з чату.

### Передумови

- Запит робить автор повідомлення **або** учасник із відповідним правом.

### Кроки

1. Користувач викликає `DELETE /chats/:chatId/messages/:messageId`.
2. Система перевіряє, чи повідомлення належить чату.
3. Система перевіряє права на видалення.
4. Система видаляє повідомлення.

### Результат

- Повідомлення більше не повертається в списку чату.

## Сценарій 6: Керування правами учасника

### Мета

Оновити набір дозволів учасника чату.

### Передумови

- Ініціатор має право `memberPermissions` (або дозволене self-оновлення відповідно до правил сервісу).

### Кроки

1. Ініціатор викликає `PATCH /chats/:chatId/members/:memberId/permissions`.
2. Система перевіряє права ініціатора.
3. Система оновлює набір прав у `member_permissions`.

### Результат

- Нові права застосовуються до наступних дій учасника в чаті.
